Day 2Week 1 · Java 21 LTS
Virtual ThreadsRecords · Sealed · Pattern Match

Java 21 — Stop Writing Java 8 in 2025

🧵 Virtual Threads — JEP 444

Write blocking code, get non-blocking throughput. Millions of threads at near-zero overhead. In Middleware: replace reactive chains with simple Executors.newVirtualThreadPerTaskExecutor() fan-outs. No WebFlux, no Mono, no Flux.

📝 Records — JEP 395

Immutable DTOs in one line. No Lombok @Data needed. Canonical constructor, equals, hashCode, toString — generated. Used for every DTO crossing a layer boundary in our architecture.

🎯 Pattern Match Switch — JEP 441

Exhaustive switch on types. Compiler enforces completeness — no unchecked branches. Domain error routing in System API becomes type-safe and compile-verified.

All 3 features — live code walkthrough
// ① Record DTO — zero boilerplate, immutable public record UserResponse(String name, String email) {} // ② Sealed class — exhaustive domain errors public sealed interface Result<T> permits Ok, Err {} public record Ok<T>(T data) implements Result<T> {} public record Err<T>(String msg) implements Result<T> {} // ③ Pattern match — compiler enforces all cases String describe(Result<?> r) { return switch (r) { case Ok<?> o -> "OK: " + o.data(); case Err<?> e -> "ERR: " + e.msg(); }; // ← compile error if case is missing } // ④ Virtual Threads — parallel fan-out is trivial try(var ex = Executors.newVirtualThreadPerTaskExecutor()) { var user = ex.submit(() -> userApi.get(id)); var prefs = ex.submit(() -> prefApi.get(id)); return new Dashboard(user.get(), prefs.get()); } // blocking .get() is fine — VTs park, not block OS thread
🔬 Day 2 Lab (12 min)
① Convert an existing DTO class (8+ fields) to a Record. ② Create a sealed Result<T> hierarchy with Ok/Err. ③ Refactor an if-instanceof chain to a pattern-match switch. Verify the compiler rejects an incomplete switch.
📋 Trainer Notes
Mid-level devs may be familiar with Lombok. Emphasise that Records are a language-level feature — no annotation processor, no byte-code manipulation. The "compiler errors on missing case" point is the key insight for sealed classes.

Day 4Week 1 · SOLID
Applied to Microservices · Not just theory

SOLID — Applied to a Distributed Architecture

S
Single Responsibility
BFF = client facing. System API = domain only. Never mix.
O
Open / Closed
Add Middleware strategies without touching stable classes.
L
Liskov Substitution
Any Repository impl must fully honour JPA contract.
I
Interface Segregation
Separate read/write ports in System API. Small interfaces.
D
Dependency Inversion
Services depend on interfaces. Spring injects concretions.
DIP violation vs correct pattern
// ❌ VIOLATION — coupled to Hibernate private HibernateUserRepositoryImpl repo; // Can't test without a real DB ↑ // ✅ CORRECT — depends on abstraction (DIP) @Service @RequiredArgsConstructor public class UserService { private final UserRepository repo; // interface! // Swap for H2 in tests — zero changes here } // ISP — split read/write ports public interface UserReadPort { UserDto find(Long id); } public interface UserWritePort { UserDto save(UserCmd cmd); } // BFF only injects UserReadPort → can't accidentally mutate
SOLID → Architecture Mapping
SBFF thin controller — delegates only, no logic
OMiddleware strategies are open to extension
LAll Repository subtypes honour same contract
ISeparate read/write ports in System API
DConstructor injection only — no @Autowired field

Day 5Week 1 · Clean Code

Clean Code — Writing for the Next Developer

📛 Names Reveal Intent
❌ Bad
d, tmp, flag, proc(), doIt()
✅ Good
daysSinceModified, findActiveUserById()
📏 Function Size Rules
  • ✦ Max 20 lines — extract if longer
  • ✦ Do exactly one thing — no side effects
  • ✦ Max 3 params — use a Parameter Object beyond that
  • ✦ No boolean flag args — split into two methods
🧹 Boy Scout Rule

Always leave code cleaner than you found it. Rename one confusing variable. Extract one long method. Fix one typo in a comment. Compound over months = maintainable codebase.

Before / After — UserService
// ❌ BEFORE — Clean Code debt public Object proc(Long id, boolean active) { Object r = repo.findById(id); if (active) { doStuff(r); } return transform(r); } // ✅ AFTER — Self-documenting, single purpose @Transactional(readOnly = true) public UserDto findActiveUserById(Long userId) { return userRepository .findByIdAndStatusActive(userId) .map(userMapper::toDto) .orElseThrow(() -> new UserNotFoundException(userId)); } // What changed: // ① Method name reveals intent (no "proc") // ② No boolean flag — active is implied by method name // ③ Optional forces caller to handle "not found" // ④ @Transactional(readOnly) signals no writes

Dockerfile — multi-stage, non-root (OWASP)
FROM eclipse-temurin:21-jdk-alpine AS build WORKDIR /app COPY pom.xml . RUN ./mvnw dependency:resolve # cache layer COPY src ./src RUN ./mvnw package -DskipTests # ── Runtime image (minimal) ── FROM eclipse-temurin:21-jre-alpine # Never run as root — OWASP A05 RUN addgroup -S capapp && adduser -S capapp -G capapp USER capapp COPY --from=build /app/target/*.jar app.jar EXPOSE 8080 ENTRYPOINT ["java", "-XX:+UseVirtualThreads", # Java 21 "-jar","app.jar"]

  • ✦ Containers talk by service name: http://middleware:8081
  • ✦ System API has no public port — only Middleware reaches it
  • ✦ BFF is the only container exposed on port 8080

Day 8Week 2 · Microservices Patterns
BFF · Aggregator · Circuit Breaker · Strangler Fig

Patterns That Make Microservices Work in Production

🎭 BFF (Backend for Frontend)

One BFF per client type. Web BFF returns rich JSON; mobile BFF returns compressed minimal payloads. Avoids over-fetching. The BFF owns the shape contract with that client — nobody else does.

🔀 Aggregator

Middleware fans out to N downstream services and merges results. Client makes one call. Correlation ID propagates the full call graph. Virtual Threads make parallel fan-out trivial — no reactive boilerplate.

🛡️ Circuit Breaker

Resilience4j monitors downstream failure rate. OPEN state stops calls when threshold exceeded. HALF-OPEN probes recovery. Prevents cascade: one failing service doesn't take down all three layers.

🌿 Strangler Fig

Incrementally strangle a monolith. BFF routes some paths to new services, others to the legacy app. No big-bang migration. Client never knows. Move one domain at a time until the monolith is empty.

📬 Transactional Outbox

Reliable event publishing: write domain event to outbox table in the same DB transaction as the domain change. Separate poller publishes to Kafka/SQS. Guarantees at-least-once delivery without 2PC.

🔑 API Gateway vs BFF

Gateway handles routing, SSL termination, rate limiting at infra level. BFF handles client-specific shaping at application level. They are complementary — a gateway sits in front of multiple BFFs in a mature deployment.

📋 Trainer Notes

Day 9Week 2 · Layer 1 BFF
Full Controller · JWT · Response Shaping

BFF — Complete Implementation

UserBffController.java — production-complete
@RestController @RequestMapping("/api/v1/users") @RequiredArgsConstructor @Validated public class UserBffController { private final MiddlewareClient mwClient; // DIP @GetMapping("/{id}") @PreAuthorize("hasRole('USER')") public UserClientView getUser( @PathVariable @Positive Long id) { // Delegate + shape — no logic in BFF return mwClient.getUser(id).toClientView(); } @ExceptionHandler(UserNotFoundException.class) public ProblemDetail notFound(UserNotFoundException e) { // OWASP A09 — structured, no stack trace return ProblemDetail.forStatusAndDetail( HttpStatus.NOT_FOUND, e.getMessage()); } } // Record DTO strips internal fields from domain object public record UserClientView( String displayName, String avatarUrl // no email, phone, internal role, or audit fields ) {}
✅ BFF Owns
  • ✦ JWT validation via @PreAuthorize
  • ✦ @Valid — input validation before any call
  • ✦ Response shaping — only fields the client needs
  • ✦ CORS headers + security response headers
  • ✦ Correlation-ID forwarding to Middleware
⛔ BFF Never Does
  • ✦ No SQL, no JPA repository calls
  • ✦ No business validation (is user eligible? — that's System API)
  • ✦ No caching of sensitive personal data
  • ✦ No service-to-service fan-out (that's Middleware)

Full JWT filter implementation is covered on Day 15. Today: understand that the filter populates SecurityContextHolder before the controller method is called. @PreAuthorize reads from that context.

🏗️ Flyway Migrations
db/migration/ V1__create_users.sql V2__add_roles.sql V3__index_email.sql V4__add_audit_cols.sql V5__soft_delete.sql

Never skip versions. Never modify existing migrations. Flyway runs at startup — schema is always current-exact.

🗄️ Repository Pattern
@Repository public interface UserRepository extends JpaRepository<User, Long> { Optional<User> findByEmail(Email email); Page<User> findByStatus( UserStatus s, Pageable p); @Modifying @Transactional @Query("UPDATE User u SET u.deletedAt=:now WHERE u.id=:id") void softDelete(Long id, Instant now); }
📐 @Transactional rules
Service layer only. readOnly=true for all reads. Never in controller or repository.
♻️ JPA Lifecycle
Transient → persist() → Managed → detach() → Detached → remove() → Removed
🐛 N+1 Problem
Use @EntityGraph or JOIN FETCH for associations. Never lazy-load in a loop.
🔒 Security (OWASP A03)
JPA parameterised queries only. Never string-concatenate SQL. @Query with named params.

JWT + Spring Security 6 — Auth at the BFF Layer

SecurityConfig.java + JwtAuthFilter.java
@Configuration @EnableMethodSecurity public class SecurityConfig { @Bean SecurityFilterChain chain(HttpSecurity h, JwtAuthFilter filter) throws Exception { return h .csrf(AbstractHttpConfigurer::disable) .sessionManagement(s -> s.sessionCreationPolicy(STATELESS)) .authorizeHttpRequests(a -> a .requestMatchers("/actuator/health").permitAll() .anyRequest().authenticated()) .addFilterBefore(filter, UsernamePasswordAuthenticationFilter.class) .build(); } } // JwtAuthFilter — intercepts every request @Component @RequiredArgsConstructor public class JwtAuthFilter extends OncePerRequestFilter { private final JwtValidator validator; @Override protected void doFilterInternal( HttpServletRequest req, HttpServletResponse res, FilterChain chain) throws Exception { extractBearer(req).ifPresent(token -> SecurityContextHolder.getContext() .setAuthentication(validator.validate(token))); chain.doFilter(req, res); } }
🔑 JWT Best Practices (OWASP A07)
  • ✦ Short access token TTL (15 min) + refresh tokens
  • ✦ RS256 (asymmetric) not HS256 in multi-service systems
  • ✦ Always validate: iss, aud, exp claims
  • ✦ Never store JWTs in localStorage — use HttpOnly cookies
  • ✦ Short TTL is your revocation mechanism — no token denylist needed
🔗 Service-to-Service Auth

BFF → Middleware and Middleware → System API use separate service JWTs. The client's token is never forwarded — re-signed with a service identity to propagate trust.

// BFF adds service JWT header client.get().uri("/users/{id}", id) .header("X-Service-Token", serviceTokenProvider.getToken()) .retrieve()...

Day 17Week 3 · Resilience & Caching
Resilience4j · Bulkhead · Redis Cache

Resilience Patterns — Failing Safely at Scale

⚡ Circuit Breaker States

Failure rate > 50%: CLOSED → OPEN. Calls immediately return fallback. After 30s: HALF-OPEN probes. Success: back to CLOSED.

🏗️ Bulkhead — Thread Isolation
resilience4j.bulkhead:   instances.userService:     maxConcurrentCalls: 25     maxWaitDuration: 100ms

Caps concurrent calls to a slow downstream. Prevents one slow service from exhausting all threads and taking down the entire Middleware.

💾 Redis Caching Strategy
@Cacheable(value="users", key="#id", unless="#result==null") public UserDto getUser(Long id) { ... } spring.cache.redis.time-to-live: 5m # Never cache: passwords, tokens, PII

Cache at Middleware layer. @CacheEvict on write. TTL ensures data freshness. Zero stale data in production.

Resilience annotation stacking order (Middleware → System API)
@RateLimiter → @Bulkhead → @CircuitBreaker → @Retry → @TimeLimiter → actual method call
🔬 Day 17 Lab (12 min)
Add @Bulkhead(name="systemApi", type=SEMAPHORE) to your orchestrator. Configure maxConcurrentCalls: 3. Use a CountDownLatch to fire 10 concurrent requests — verify 7 get BulkheadFullException. Add Redis cache with 2-min TTL to getUser. Confirm second call is served from cache (check logs).
📋 Trainer Notes
Mid-level devs rarely think about Bulkhead patterns — they know Circuit Breakers from theory but not the thread isolation angle. The concurrent-request demo is very effective. Use a simple loop rather than a test framework — they can see the exceptions live in the console.
📊 Exit Check (5 pts)
✦ Bulkhead configured and rejects concurrent excess (2 pts) ✦ Resilience annotations in correct order (1 pt) ✦ Redis cache returns hit on second call (no downstream call) (2 pts)

@Transactional Boundaries · Domain Events · ApplicationEventPublisher

Transactions & Domain Events — Correctness at the Core

UserService — correct transaction boundaries + domain event
@Service @RequiredArgsConstructor public class UserService { private final UserRepository repo; private final ApplicationEventPublisher events; @Transactional // ← service layer only public UserDto deactivateUser(Long userId) { var user = repo.findById(userId) .orElseThrow(() -> new UserNotFoundException(userId)); user.deactivate(); // domain method raises event internally repo.save(user); // explicit save — don't rely on dirty check // Publish AFTER commit — not inside transaction! events.publishEvent(new UserDeactivatedEvent(userId)); return userMapper.toDto(user); } @Transactional(readOnly = true) // ← 20-30% faster public Page<UserDto> findAll(Pageable p) { return repo.findAll(p).map(userMapper::toDto); } }
📐 @Transactional Rules for Mid-Level Devs
  • ✦ Service layer only — never controller, never repository
  • ✦ readOnly=true on all query methods — performance + correctness
  • ✦ Keep transactions short — never make HTTP calls inside a transaction
  • ✦ propagation=REQUIRES_NEW for audit logging that must survive rollback
📢 @TransactionalEventListener
@Component public class UserDeactivatedHandler { @TransactionalEventListener // ← fires AFTER commit (not before!) public void handle(UserDeactivatedEvent e) { auditService.log(e.userId(), "deactivated"); // Safe: even if audit fails, deactivation committed } }

Day 20Week 3 · Service Communication
RestClient · Error Propagation · Correlation IDs

Inter-Service Communication — Contracts & Error Propagation

MiddlewareClient.java — error-aware RestClient
@Component @RequiredArgsConstructor public class MiddlewareClient { private final RestClient client; public UserDto getUser(Long id) { return client.get() .uri("/api/v1/users/{id}", id) .header("X-Correlation-Id", correlationProvider.current()) .retrieve() .onStatus(HttpStatusCode::is4xxClientError, (req, res) -> { throw new UserNotFoundException(id); }) .onStatus(HttpStatusCode::is5xxServerError, (req, res) -> { throw new DownstreamServiceException("middleware"); }) .body(UserDto.class); } } // Correlation-ID request interceptor @Bean RestClient restClient(RestClient.Builder b) { return b.requestInterceptor((req, body, ex) -> { req.getHeaders().set("X-Correlation-Id", MDC.get("traceId")); // propagate trace return ex.execute(req, body); }).build(); }
🔄 Error Translation Strategy
Downstream returnsClient exception
404UserNotFoundException → BFF returns 404
503DownstreamServiceException → fallback or 503
500Circuit opens after threshold
🏷️ Correlation-ID Rules
  • ✦ Generated at BFF entry point if not present in request
  • ✦ Stored in MDC for all log lines in that thread
  • ✦ Forwarded via X-Correlation-Id header to downstream
  • ✦ Every log line automatically includes it via Micrometer
💡 Mid-Level Insight

Don't let HTTP status codes leak raw internal errors to clients. Translate: System API 422 → BFF 422 with clean message. System API 500 → BFF 503 with generic "service unavailable". Client never sees internal stack traces.

📋 Trainer Notes
The correlation ID propagation is the single most useful debugging tool mid-level devs will ever add. Show the before (finding a bug in 3 log streams) vs after (grep for one traceId to get the complete picture). Time the difference — it's dramatic.

Day 22Week 4 · Security
OWASP Top 10 — Mapped to Our Architecture

OWASP Top 10 — Mitigated by Architecture

A01Broken Access Control

BFF: @PreAuthorize on every endpoint. System API: no public port — network-isolated. Least-privilege DB user: SELECT/INSERT only.

A02Crypto Failures

HTTPS-only at BFF (TLS 1.3). DATABASE_URL in env secret. No PII in Redis. At-rest encryption at cloud level.

A03Injection

JPA parameterised queries only. @Valid + Bean Validation at BFF entry. @SafeHtml on text inputs. Never string-built SQL.

A05Misconfiguration

CORS strictly configured. Actuator secured. Containers run as non-root. No debug endpoints or stack traces in prod responses.

A07Auth Failures

JWT RS256 with 15-min TTL. Stateless sessions. Spring Security 6 filter chain. Service-to-service JWT with short TTL.

A09Logging Failures

Structured JSON logs. Correlation IDs. Never log passwords, tokens, PII. ProblemDetail error format — no stack traces to clients.

Defence-in-Depth: Each layer validates independently. A BFF breach doesn't automatically compromise System API — network isolation, input validation, and least-privilege DB access form independent perimeters. The architecture itself is the security control.

🔬 Day 22 Lab (12 min)

📋 Trainer Notes
The most effective demo: show them how to extract the full stack trace from a Spring error response (before ProblemDetail). Then show what an attacker can learn from it (class names, framework versions, internal paths). The shift from abstract to concrete makes OWASP A09 land immediately.
