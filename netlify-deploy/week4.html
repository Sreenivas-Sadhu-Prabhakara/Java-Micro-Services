DAY 16 Phase 4 — BFF Gateway

Spring Cloud Gateway — Routing & Filters

📥 Input
Day 15 — System API + Middleware fully operational BFF module has basic Spring Boot scaffold only
🔧 Activities
  1. Replace spring-boot-starter-web with gateway — BFF uses spring-cloud-starter-gateway (reactive/Netty). Remove spring-boot-starter-web (blocking/Tomcat). They conflict.
  2. Define routes in YAML — Route /api/v1/products/** → middleware:8082. Route /api/v1/system/** → system-api:8081 (for admin). Use predicates: Path, Method, Header.
  3. Add gateway filters — AddRequestHeader, AddResponseHeader, StripPrefix, RewritePath. Add request/response logging filter.
  4. Create custom GlobalFilter — Log every request: method, path, duration. Add X-Response-Time header to every response.
  5. Configure CORS — Allow specific origins, methods, headers. Critical for browser-based clients calling the BFF.
  6. Test routing — curl localhost:8080/api/v1/products → routed to middleware → routed to system-api. Full chain through BFF.
  7. Add Actuator gateway routes — /actuator/gateway/routes to inspect all configured routes and filters at runtime.
📐 Principles
  • API Gateway Pattern — Single entry point. Cross-cutting concerns (auth, logging, CORS) centralized.
  • Reactive — Gateway uses Spring WebFlux/Netty. Non-blocking I/O. High throughput.
  • Defense in Depth — Only BFF port exposed. Internal services unreachable from outside.

          filters:
            - RewritePath=/admin(?<segment>/?.*), $\{segment}
💻 Custom Global Filter
@Component
@Slf4j
public class RequestTimingFilter implements GlobalFilter, Ordered {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange,
                              GatewayFilterChain chain) {
        long start = System.currentTimeMillis();
        String path = exchange.getRequest().getPath().value();
        return chain.filter(exchange).then(Mono.fromRunnable(() -> {
            long ms = System.currentTimeMillis() - start;
            exchange.getResponse().getHeaders()
                .add("X-Response-Time", ms + "ms");
            log.info("Gateway: {} {} → {}ms", 
                exchange.getRequest().getMethod(), path, ms);
        }));
    }

    @Override public int getOrder() { return -1; }
}

DAY 17 Phase 4 — BFF Gateway

OAuth2 / JWT Security & Authentication

📥 Input
Day 16 — BFF Gateway routing traffic, no security All endpoints publicly accessible
🔧 Activities
  1. Add OAuth2 Resource Server — spring-boot-starter-oauth2-resource-server in BFF. Gateway validates JWT tokens on every request before routing.
  2. Setup Keycloak (or mock) — Run Keycloak in Docker for dev. Create realm, client, users. Alternatively use a JWT mock for testing.
  3. Configure JWT validation — Set spring.security.oauth2.resourceserver.jwt.issuer-uri to Keycloak. Gateway validates signature, expiry, issuer, audience.
  4. Create SecurityConfig — Permit /actuator/health, protect all /api/** routes. Define role-based access: ROLE_USER for read, ROLE_ADMIN for write.
  5. Propagate JWT downstream — Pass Authorization header from BFF to middleware. Middleware can extract user info from token for audit logging.
  6. Add CSRF protection — Disable for stateless API (JWT). Document why. Enable for cookie-based flows if needed.
  7. Test security — Call without token → 401. Call with expired token → 401. Call with valid token → 200. Call with wrong role → 403.
📐 Principles
  • OWASP A01 — Broken Access Control prevented via JWT + role-based authorization.
  • OWASP A07 — Authentication failures handled. Rate limiting on auth endpoints.
  • Zero Trust — Validate token at gateway. Don't trust internal services blindly.
💻 Security Config
@Configuration
@EnableWebFluxSecurity
public class SecurityConfig {

    @Bean
    SecurityWebFilterChain securityFilterChain(
            ServerHttpSecurity http) {
        return http
            .csrf(ServerHttpSecurity.CsrfSpec::disable)
            .authorizeExchange(auth -> auth
                .pathMatchers("/actuator/**").permitAll()
                .pathMatchers(GET, "/api/v1/**").hasRole("USER")
                .pathMatchers(POST, "/api/v1/**").hasRole("ADMIN")
                .pathMatchers(PUT, "/api/v1/**").hasRole("ADMIN")
                .pathMatchers(DELETE, "/api/v1/**").hasRole("ADMIN")
                .anyExchange().authenticated())
            .oauth2ResourceServer(oauth2 -> oauth2
                .jwt(jwt -> jwt
                    .jwtAuthenticationConverter(
                        jwtAuthConverter())))
            .build();
    }
}

DAY 18 Phase 4 — BFF Gateway

Rate Limiting, CORS & Gateway Hardening

📥 Input
Day 17 — OAuth2/JWT auth working on BFF Gateway No rate limiting, no CORS policy, no abuse protection
🔧 Activities
  1. Add RequestRateLimiter filter — Integrate spring-cloud-gateway rate limiter with Redis. Configure replenishRate, burstCapacity, and requestedTokens per route.
  2. Redis for token bucket — Add Redis container to docker-compose. Gateway stores rate-limit counters in Redis. Test with k6 or wrk to verify throttling.
  3. Configure CORS globally — Set allowed origins, methods, headers in Gateway. Use CorsWebFilter bean. Allow Authorization header for JWT propagation.
  4. Request/Response logging filter — Create custom GlobalFilter to log request method, path, status, latency. Mask sensitive headers (Authorization). Correlation ID in MDC.
  5. Add response headers — X-Content-Type-Options: nosniff, X-Frame-Options: DENY, Strict-Transport-Security, Cache-Control on sensitive endpoints.
  6. Request size limits — Configure max-in-memory-size to prevent large payload attacks. Return 413 Payload Too Large for oversized requests.
  7. Integration tests — Test rate limiting returns 429. Test CORS preflight returns correct headers. Test oversized payloads rejected.
📐 Principles
  • OWASP A05 — Security Misconfiguration prevented via strict CORS, security headers.
  • Throttle at the edge — Rate limiting at gateway protects all downstream services.
  • Defense in depth — Multiple layers: auth + rate limit + CORS + headers.

💻 CORS & Security Headers
@Bean
CorsWebFilter corsFilter() {
    CorsConfiguration config = new CorsConfiguration();
    config.addAllowedOrigin("http://localhost:3000");
    config.addAllowedMethod("*");
    config.addAllowedHeader("*");
    config.setAllowCredentials(true);
    config.setMaxAge(3600L);

    UrlBasedCorsConfigurationSource src =
        new UrlBasedCorsConfigurationSource();
    src.registerCorsConfiguration("/api/**", config);
    return new CorsWebFilter(src);
}

@Bean
KeyResolver userKeyResolver() {
    return exchange -> Mono.justOrEmpty(
        exchange.getRequest().getHeaders()
            .getFirst("X-Forwarded-For"))
        .defaultIfEmpty(exchange.getRequest()
            .getRemoteAddress().getHostString());
}

DAY 19 Phase 5 — Observability

Centralized Logging — ELK Stack & Structured JSON Logs

📥 Input
Day 18 — Hardened BFF Gateway with request logging Logs scattered across containers, no aggregation
🔧 Activities
  1. Switch to Logback JSON encoder — Add logstash-logback-encoder dependency. Configure logback-spring.xml with LogstashEncoder for structured JSON output.
  2. Add MDC fields — Inject traceId, spanId, serviceName, correlationId into every log line. Use Micrometer Tracing for auto-propagation.
  3. Deploy ELK in Docker Compose — Elasticsearch, Logstash, Kibana containers. Logstash receives JSON logs via TCP appender. Elasticsearch indexes. Kibana visualizes.
  4. Configure Logstash pipeline — Parse JSON logs, add metadata fields (environment, version), route to Elasticsearch index per service per day.
  5. Kibana dashboards — Create index patterns. Build dashboards: error rate by service, latency percentiles, log volume over time, top error messages.
  6. Log levels per environment — DEBUG in dev, INFO in staging, WARN in prod. Use Spring profiles to switch. Runtime level change via actuator.
  7. Sensitive data masking — Mask PII (emails, tokens) in logs using Logback pattern replacements. Never log passwords or full credit card numbers.
📐 Principles
  • 12-Factor XI — Treat logs as event streams. Write to stdout, ship externally.
  • Observability Pillar #1 — Logs are the narrative. Structured + correlated = debuggable.
  • GDPR compliance — Mask PII in logs. Retention policies on Elasticsearch indices.

💻 Custom Metrics
@Component
@RequiredArgsConstructor
public class OrderMetrics {

    private final MeterRegistry registry;

    private Counter ordersCreated;
    private Counter ordersFailed;
    private Timer   orderProcessing;

    @PostConstruct
    void init() {
        ordersCreated = Counter.builder(
                "orders.created.total")
            .description("Total orders created")
            .tag("service", "order-service")
            .register(registry);

        ordersFailed = Counter.builder(
                "orders.failed.total")
            .tag("service", "order-service")
            .register(registry);

        orderProcessing = Timer.builder(
                "order.processing.duration")
            .publishPercentiles(0.5, 0.95, 0.99)
            .register(registry);
    }

    public void recordCreated() {
        ordersCreated.increment();
    }

    public Timer.Sample startTimer() {
        return Timer.start(registry);
    }
}
