DAY 26 Phase 7 — CI/CD & Deployment

Kubernetes Manifests — Deployments, Services & ConfigMaps

📥 Input
Day 25 — CI pipeline producing Docker images to registry Running only in Docker Compose, no orchestration
🔧 Activities
  1. Setup local K8s — Install Minikube or kind. Create namespace microservices-dev. Configure kubectl context. Enable ingress addon.
  2. Deployment manifests — Create k8s/ directory. Write Deployment for each service: replicas, resource limits (CPU/memory), image pull policy, env vars from ConfigMap/Secret.
  3. Service manifests — ClusterIP Service for internal communication. NodePort/Ingress for BFF Gateway external access. Service DNS: order-service.microservices-dev.svc.cluster.local.
  4. ConfigMaps & Secrets — ConfigMap for Spring profiles, feature flags, service URLs. Secrets for DB passwords, JWT keys, API tokens. Mount as env vars or volumes.
  5. Health probes in K8s — Map Spring Boot actuator probes to K8s: livenessProbe → /actuator/health/liveness, readinessProbe → /actuator/health/readiness, startupProbe.
  6. Resource quotas — Set requests and limits: requests: {cpu: 250m, memory: 512Mi}, limits: {cpu: 1000m, memory: 1Gi}. Prevent resource starvation.
  7. Deploy & verify — kubectl apply -k k8s/. Verify pods running, services reachable, health probes passing, inter-service communication working.
📐 Principles
  • Declarative infrastructure — Desired state in YAML, K8s reconciles. GitOps ready.
  • 12-Factor III — Store config in environment, not in code. ConfigMaps/Secrets enable this.
  • Resource governance — Limits prevent noisy neighbors. Requests ensure scheduling fairness.

DAY 27 Phase 7 — CI/CD & Deployment

Helm Charts, GitOps & CD Pipeline

📥 Input
Day 26 — Raw K8s manifests deployed manually with kubectl No templating, no environment promotion, no automated CD
🔧 Activities
  1. Create Helm chart — helm create microservices. Templatize Deployments, Services, ConfigMaps. Use values.yaml for image tags, replica counts, resource limits per environment.
  2. Values per environment — values-dev.yaml (1 replica, debug logs), values-staging.yaml (2 replicas, info logs), values-prod.yaml (3 replicas, warn logs, HPA enabled).
  3. Helm dependencies — Package PostgreSQL, Redis, Kafka as chart dependencies. Pin versions. Use condition flags to toggle infrastructure in dev vs prod (prod uses managed services).
  4. CD pipeline with GitHub Actions — On merge to main → build images → update Helm values with new image tag → helm upgrade --install to staging. Manual approval gate for prod.
  5. GitOps with ArgoCD (intro) — Install ArgoCD in K8s. Point to Git repo containing Helm charts. ArgoCD syncs desired state from Git to cluster automatically.
  6. Rollback strategy — Helm rollback: helm rollback microservices 1. Keep 5 revision history. Blue-green and canary deployment concepts discussed.
  7. Helm test hooks — Create test pod that runs smoke tests after deployment. helm test microservices verifies health endpoints and basic API calls post-deploy.
📐 Principles
  • GitOps — Git is the single source of truth. All changes through PRs. Cluster state = Git state.
  • Environment parity — Same chart, different values. Dev ≈ staging ≈ prod topology.
  • Immutable deployments — Never patch running containers. Always deploy new image + config.

DAY 28 Phase 8 — Production Readiness

Performance Testing — k6 Load Tests & Optimization

📥 Input
Day 27 — Full CI/CD pipeline deploying to K8s No performance baselines, unknown capacity limits
🔧 Activities
  1. Install k6 & write test scripts — JavaScript-based load test scenarios. Test order creation flow: authenticate → create order → verify response. Parameterize with test data from CSV.
  2. Smoke test (1 VU) — Single virtual user, verify correctness under no load. Establish baseline response times and validate test script logic.
  3. Load test (50 VUs, 5 min) — Simulate normal traffic. Measure p50/p95/p99 latency, throughput (RPS), error rate. Set SLA thresholds: p99 < 500ms, error rate < 1%.
  4. Stress test (ramp to 200 VUs) — Find breaking point. Ramp up gradually: 10 → 50 → 100 → 200 VUs. Monitor where latency spikes and errors increase. Identify bottleneck.
  5. Bottleneck analysis — Correlate k6 results with Grafana dashboards. Check: connection pool exhaustion? Thread starvation? GC pauses? DB query slow? Kafka consumer lag?
  6. Optimization cycle — Add database indices, tune connection pool sizes (hikari.maximum-pool-size), enable response compression, add Redis cache for hot paths. Re-test.
  7. Performance CI gate — Add k6 to CI pipeline. Run smoke test on every PR. Run full load test nightly. Fail if SLA thresholds breached. Track trends over time.
📐 Principles
  • Test early, test often — Performance is a feature. Regressions caught by automated load tests in CI.
  • Little's Law — L = λW (concurrency = throughput × latency). Understand system capacity mathematically.
  • Amdahl's Law — Optimize the bottleneck first. Profiling before guessing.
💻 k6 Load Test Script
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate, Trend } from 'k6/metrics';

const errorRate = new Rate('errors');
const orderLatency = new Trend('order_latency');

export const options = {
  stages: [
    { duration: '1m', target: 10  },
    { duration: '3m', target: 50  },
    { duration: '2m', target: 100 },
    { duration: '1m', target: 0   },
  ],
  thresholds: {
    http_req_duration: ['p(99)<500'],
    errors: ['rate<0.01'],
  },
};

const BASE = 'http://bff-gateway:8080';

export default function () {
  // 1. Authenticate
  const auth = http.post(
    `${BASE}/auth/token`,
    JSON.stringify({
      username: 'testuser',
      password: 'testpass' }),
    { headers:
      { 'Content-Type': 'application/json' }}
  );
  const token = auth.json('access_token');

  // 2. Create Order
  const start = Date.now();
  const res = http.post(
    `${BASE}/api/v1/orders`,
    JSON.stringify({
      customerId: 'cust-001',
      items: [{ sku: 'SKU-100', qty: 2 }]
    }),
    { headers: {
        'Content-Type': 'application/json',
        Authorization: `Bearer ${token}`
    }}
  );
  orderLatency.add(Date.now() - start);

  check(res, {
    'status 201': (r) => r.status === 201,
    'has orderId': (r) =>
      r.json('orderId') !== undefined,
  }) || errorRate.add(1);

  sleep(1);
}

DAY 29 Phase 8 — Production Readiness

Contract Testing, E2E Testing & Quality Assurance

📥 Input
Day 28 — Performance tested, SLA baselines established No contract tests between services, no full E2E suite
🔧 Activities
  1. Spring Cloud Contract — Provider side — Write contracts in Groovy DSL for order-service API. Auto-generate tests that verify provider meets contract. Auto-generate WireMock stubs for consumers.
  2. Consumer-Driven Contracts — Middleware declares what it expects from order-service. Provider verifies it can fulfill. Break a contract → CI fails before deployment.
  3. Pact alternative (overview) — Compare Spring Cloud Contract vs Pact. Pact for polyglot environments. Pact Broker for contract sharing. Choose based on team structure.
  4. Testcontainers E2E — Full integration test: spin up all services + Kafka + Postgres + Redis in Testcontainers. Create order → verify event flow → check inventory → assert final state.
  5. Test pyramid validation — Count tests: ~200 unit tests (fast, isolated), ~50 integration tests (Spring context), ~10 contract tests (API boundaries), ~5 E2E tests (full flow). Verify ratio is healthy.
  6. Mutation testing — Run PITest to verify test quality. If mutating code doesn't fail tests → tests are weak. Target > 70% mutation score on domain logic.
  7. Test reporting — Aggregate JaCoCo + contract + E2E + mutation reports. Single quality dashboard. All green = production-ready confidence.
📐 Principles
  • Test Pyramid — Many unit tests, fewer integration, fewest E2E. Fast feedback at the base.
  • Consumer-Driven Contracts — APIs evolve safely. Breaking changes caught at build time, not runtime.
  • Mutation testing — Tests that don't catch bugs are false confidence. Mutants measure test effectiveness.

💻 ADR Template
# ADR-001: Hexagonal Architecture
## Status: Accepted
## Context
We need a service architecture that
isolates business logic from infrastructure
to enable testability, portability, and
independent evolution of adapters.

## Decision
Adopt Hexagonal (Ports & Adapters) for
all microservices. Domain logic in core
module. REST/Kafka/DB as adapters.

## Consequences
+ Business logic testable without Spring
+ Swappable adapters (REST → gRPC)
+ Enforced by ArchUnit in CI
- Steeper learning curve initially
- More packages/classes than layered
