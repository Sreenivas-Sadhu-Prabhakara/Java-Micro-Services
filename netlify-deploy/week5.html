DAY 21 Phase 5 — Observability

Distributed Tracing — OpenTelemetry & Zipkin/Jaeger

📥 Input
Day 20 — Metrics + logging operational with correlation IDs Cannot visualize request flow across services
🔧 Activities
  1. Add Micrometer Tracing + Bridge — micrometer-tracing-bridge-otel and opentelemetry-exporter-zipkin. Auto-instruments RestClient, WebClient, JPA, messaging.
  2. Deploy Zipkin — Docker container on port 9411. All services export spans to Zipkin. Alternatively use Jaeger with OTLP collector for production parity.
  3. Configure sampling — 100% in dev, 10% in prod via management.tracing.sampling.probability. Balance observability cost vs coverage.
  4. Trace context propagation — W3C TraceContext headers propagated automatically. Verify: BFF → Middleware → Backend shows single trace with multiple spans.
  5. Custom spans — Add @Observed annotation on key business methods. Create custom spans for external API calls, cache lookups, complex computations.
  6. Baggage propagation — Pass userId, tenantId as baggage items across services. Available in all downstream logs without explicit passing.
  7. Trace-to-log correlation — Click trace in Zipkin → link to Kibana logs with same traceId. Full request narrative: trace + logs + metrics.
📐 Principles
  • Observability Pillar #3 — Traces show the journey. Combined with logs and metrics = complete picture.
  • OpenTelemetry standard — Vendor-neutral instrumentation. Switch backends without code changes.
  • Sampling strategy — Head-based vs tail-based. Always sample errors and slow requests.

DAY 22 Phase 5 — Observability

Health Checks, Readiness Probes & Observability Checkpoint

📥 Input
Day 21 — Full observability: logs + metrics + traces integrated No custom health indicators, no Kubernetes-ready probes
🔧 Activities
  1. Custom HealthIndicators — Implement HealthIndicator for external dependencies: database connectivity, Redis availability, Keycloak reachability, disk space.
  2. Liveness vs Readiness — /actuator/health/liveness = app alive (not deadlocked). /actuator/health/readiness = app ready (DB connected, caches warm). Configure Kubernetes probe groups.
  3. Startup probe — Slow-starting services get /actuator/health/startup probe. Prevents premature restarts. Configure initialDelaySeconds, periodSeconds.
  4. Graceful shutdown — Enable server.shutdown=graceful. In-flight requests complete before shutdown. Deregister from service discovery first. 30s timeout.
  5. Observability smoke test — End-to-end: create order → verify log in Kibana → verify trace in Zipkin → verify metric in Grafana → verify health endpoint green.
  6. Runbook documentation — Create runbook: what each alert means, how to investigate, escalation path. Link from Grafana alert to runbook page.
  7. Phase 5 checkpoint — All 3 observability pillars functional. Team demo: trace a failed request through logs, traces, and metrics to root cause.
📐 Principles
  • Cloud-Native Health — Probes enable self-healing. Kubernetes restarts unhealthy pods, stops routing to unready pods.
  • Graceful degradation — Shutdown without dropping requests. Zero-downtime deployments depend on this.
  • MTTR over MTBF — Invest in fast recovery (observability) over preventing all failures.
💻 Custom Health Indicator
@Component
public class KeycloakHealthIndicator
        implements HealthIndicator {

    private final RestClient restClient;

    public KeycloakHealthIndicator(
            RestClient.Builder builder) {
        this.restClient = builder
            .baseUrl("http://keycloak:8080")
            .build();
    }

    @Override
    public Health health() {
        try {
            restClient.get()
                .uri("/realms/master")
                .retrieve()
                .toBodilessEntity();
            return Health.up()
                .withDetail("realm", "reachable")
                .build();
        } catch (Exception e) {
            return Health.down(e)
                .withDetail("realm", "unreachable")
                .build();
        }
    }
}

DAY 23 Phase 6 — Event-Driven & Async

Apache Kafka Setup & Event-Driven Architecture

📥 Input
Day 22 — Full observability stack operational, all sync HTTP calls Tight coupling: order-service directly calls inventory-service
🔧 Activities
  1. Deploy Kafka + Zookeeper — Add Kafka (KRaft mode) + Kafka UI to docker-compose. Create topics: order-events, inventory-events, notification-events with 3 partitions.
  2. Define domain events — OrderCreatedEvent, OrderCancelledEvent, InventoryReservedEvent, InventoryFailedEvent. Use CloudEvents spec for envelope. Avro or JSON schema.
  3. Spring Kafka producer — Configure KafkaTemplate with JSON serializer. Publish events from domain service after successful DB transaction. Use @TransactionalEventListener for consistency.
  4. Spring Kafka consumer — @KafkaListener on inventory-service listens to order-events. Processes OrderCreatedEvent → reserves stock → publishes InventoryReservedEvent.
  5. Idempotent consumers — Store processed event IDs in DB. Skip duplicates. Use eventId (UUID) as deduplication key. Handle at-least-once delivery safely.
  6. Dead letter topic — Configure DLT for failed messages. After 3 retries → send to order-events.DLT. Monitor DLT size in Grafana.
  7. Kafka UI exploration — Use Kafka UI to inspect topics, partitions, consumer groups, lag. Verify messages flowing correctly.
📐 Principles
  • Event-Driven Architecture — Services communicate via events, not direct calls. Loose coupling, temporal decoupling.
  • At-least-once delivery — Kafka guarantees delivery but may duplicate. Consumer idempotency is mandatory.
  • Event Sourcing foundation — Events as the source of truth. Enables replay, audit, CQRS patterns.
💻 Event Producer
@Service
@RequiredArgsConstructor
public class OrderEventPublisher {

    private final KafkaTemplate<String, Object> kafka;

    @TransactionalEventListener(
        phase = AFTER_COMMIT)
    public void onOrderCreated(
            OrderCreatedEvent event) {
        kafka.send("order-events",
            event.orderId().toString(),
            CloudEvent.builder()
                .id(UUID.randomUUID().toString())
                .type("order.created.v1")
                .source("/order-service")
                .data(event)
                .build());
    }
}
💻 Event Consumer
@Component
@RequiredArgsConstructor
public class OrderEventConsumer {

    private final InventoryService inventory;
    private final ProcessedEventRepo processed;

    @KafkaListener(
        topics = "order-events",
        groupId = "inventory-service")
    public void handle(
            @Payload CloudEvent event,
            @Header(KafkaHeaders.RECEIVED_KEY)
                String key) {

        if (processed.existsById(event.id()))
            return; // idempotent skip

        var cmd = parse(event);
        inventory.reserveStock(cmd);
        processed.save(event.id());
    }
}

DAY 24 Phase 6 — Event-Driven & Async

Saga Pattern — Orchestrated Distributed Transactions

📥 Input
Day 23 — Kafka events flowing between services No compensation logic when downstream steps fail
🔧 Activities
  1. Design order saga — Steps: Create Order → Reserve Inventory → Process Payment → Confirm Order. Each step has a compensating action (Cancel Order, Release Inventory, Refund Payment).
  2. Saga orchestrator — Build OrderSagaOrchestrator in middleware. Listens for events, decides next step. Maintains saga state in DB (STARTED, INVENTORY_RESERVED, PAYMENT_PROCESSED, COMPLETED, COMPENSATING, FAILED).
  3. Compensation flow — If payment fails → publish ReleaseInventoryCommand → publish CancelOrderCommand. Each service handles its own rollback. Eventually consistent.
  4. Saga state machine — Use Spring State Machine or simple enum-based FSM. Track state transitions. Log every transition for audit.
  5. Timeout handling — If no response within 30s → trigger compensation. Use Kafka consumer timeout + scheduled job to check stale sagas.
  6. Testing sagas — Happy path: all steps succeed. Failure at step 2: compensation triggered. Failure at compensation: alert + manual intervention queue.
  7. Saga observability — Custom metrics: saga_started, saga_completed, saga_compensated. Trace spans for each saga step. Dashboard showing saga success rate.
📐 Principles
  • BASE over ACID — Basically Available, Soft state, Eventually consistent. No distributed locks.
  • Compensating transactions — Every action must have an undo. Design for failure from the start.
  • Orchestration vs Choreography — Orchestrator = centralized control. Choreography = each service reacts. We use orchestration for complex multi-step flows.
💻 Saga Orchestrator
@Service
@RequiredArgsConstructor
public class OrderSagaOrchestrator {

    private final SagaStateRepo repo;
    private final KafkaTemplate<String, Object> kafka;

    @KafkaListener(topics = "order-events")
    public void onOrderCreated(OrderCreatedEvent e) {
        var saga = SagaState.start(e.orderId());
        repo.save(saga);
        kafka.send("inventory-commands",
            new ReserveStockCmd(e.orderId(),
                e.items()));
        saga.transition(INVENTORY_PENDING);
        repo.save(saga);
    }

    @KafkaListener(topics = "inventory-events")
    public void onInventoryResult(
            InventoryEvent event) {
        var saga = repo.findByOrderId(
            event.orderId());
        if (event instanceof InventoryReserved) {
            saga.transition(INVENTORY_OK);
            kafka.send("payment-commands",
                new ProcessPaymentCmd(
                    saga.orderId(), saga.total()));
        } else {
            saga.transition(COMPENSATING);
            kafka.send("order-commands",
                new CancelOrderCmd(saga.orderId()));
        }
        repo.save(saga);
    }
}
💻 Saga State Entity
@Entity @Table(name = "saga_state")
public class SagaState {
    @Id
    private UUID sagaId;
    private UUID orderId;

    @Enumerated(EnumType.STRING)
    private SagaStatus status;

    private Instant startedAt;
    private Instant completedAt;

    @Version
    private int version; // optimistic lock

    public enum SagaStatus {
        STARTED, INVENTORY_PENDING,
        INVENTORY_OK, PAYMENT_PENDING,
        COMPLETED, COMPENSATING, FAILED
    }

    public void transition(SagaStatus next) {
        log.info("Saga {} : {} → {}",
            sagaId, status, next);
        this.status = next;
    }
}

DAY 25 Phase 7 — CI/CD & Deployment

GitHub Actions CI — Build, Test & Publish Pipeline

📥 Input
Day 24 — Complete microservices app with sync + async communication No CI pipeline, manual builds, no automated quality gates
🔧 Activities
  1. Create GitHub Actions workflow — .github/workflows/ci.yml triggered on push/PR to main. Matrix build: JDK 21. Cache Maven dependencies for speed.
  2. Build stage — mvn verify -B compiles all modules, runs unit tests, runs integration tests. Fail pipeline on any test failure. Publish test results as annotations.
  3. Code quality stage — Run Checkstyle, SpotBugs, JaCoCo coverage. Minimum 80% line coverage gate. Upload coverage report to Codecov or SonarCloud.
  4. Container build stage — Build Docker images with Jib (mvn jib:build). Tag with git SHA + latest. Push to GitHub Container Registry (ghcr.io).
  5. Security scanning — Trivy scan on Docker images for CVEs. OWASP Dependency-Check on Maven dependencies. Fail on CRITICAL/HIGH vulnerabilities.
  6. PR checks — Require CI pass before merge. Auto-label PRs by changed path. Run only affected module tests on PR (build caching).
  7. Artifact management — Publish JARs to GitHub Packages. Docker images versioned with semver tags. Retention policy for old artifacts.
📐 Principles
  • CI = Fast Feedback — Every push validates in < 10 minutes. Developers know immediately if they broke something.
  • Shift Left Security — Scan dependencies and containers in CI, not after deployment.
  • Trunk-based development — Short-lived branches, frequent merges to main. CI enforces quality.
