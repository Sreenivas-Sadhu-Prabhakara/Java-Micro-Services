Strategic Context

Why Microservices? — SCQA Framework

Situation

Enterprise clients require systems that handle high concurrency, deploy independently, and scale components in isolation. Teams must ship features continuously with zero downtime across global regions.

Complication

Monolithic Java apps cannot scale individual features. A single bug crashes the entire system. Tight coupling means every change is risky. Deployment windows are weeks, not minutes. New engineers take months to onboard.

Question

How do we build a greenfield system from Day 1 that is independently deployable, fault-tolerant, secured at every layer, and observable — using Java 21 and open-source tooling?

Answer

A 3-layer microservices architecture — BFF → Middleware → System API — each in its own Docker container, connected to PostgreSQL via JPA, built with Java 21, secured with OWASP, following SOLID/Clean Code.

Architecture Blueprint

Target 3-Layer Distributed System

L1

BFF Gateway

Purpose: The single entry point for all client traffic. Shields internal services from direct exposure.

What to Expect:
  • Route definitions mapping URL paths to downstream services
  • JWT token validation on every incoming request
  • Rate limiting per client (token bucket algorithm)
  • Response aggregation — combine data from multiple services into a single client payload
  • CORS/CSRF configuration for browser security
How You'll Build It:
  • Start with spring-cloud-starter-gateway
  • Define routes in application.yml using predicates & filters
  • Add spring-boot-starter-oauth2-resource-server for JWT
  • Configure RequestRateLimiter filter with Redis backing
  • Write custom GatewayFilter for header manipulation
Days 16–20 Spring Cloud Gateway
L2

Middleware

Purpose: Business orchestration layer. Composes calls to System API, applies business rules, transforms data, handles failures.

What to Expect:
  • Declarative REST clients via OpenFeign to call System API
  • Service composition — orchestrate multiple downstream calls
  • DTO ↔ Entity mapping using MapStruct or manual mappers
  • Circuit breakers (Resilience4j) for fault tolerance
  • Global exception handling with RFC 7807 Problem Detail
  • Virtual Threads (Project Loom) for massive concurrency
How You'll Build It:
  • Add spring-cloud-starter-openfeign dependency
  • Create @FeignClient interfaces pointing to system-api
  • Implement @Service classes that compose Feign calls
  • Add @CircuitBreaker annotations with fallback methods
  • Create @RestControllerAdvice for centralized error handling
Days 11–15 OpenFeign + Resilience4j
L3

System API

Purpose: Data access layer. Owns the database schema, JPA entities, and exposes CRUD REST endpoints. The single source of truth for data.

What to Expect:
  • PostgreSQL running in Docker with persistent volumes
  • JPA entities with proper relationships (OneToMany, ManyToOne)
  • Flyway migrations for version-controlled schema changes
  • Spring Data JPA repositories with custom queries
  • Bean Validation (@Valid) on all incoming DTOs
  • SpringDoc OpenAPI for auto-generated Swagger docs
How You'll Build It:
  • Add PostgreSQL container in docker-compose.yml
  • Create @Entity classes with JPA annotations
  • Write Flyway SQL migrations in db/migration/
  • Extend JpaRepository<T, ID> for each entity
  • Build @RestController with full CRUD (GET/POST/PUT/DELETE)
  • Add springdoc-openapi-starter-webmvc-ui for Swagger
Days 6–10 JPA + Flyway + OpenAPI

🔗 Inter-Service Communication Patterns

①
Client → BFF: HTTPS/TLS. JSON payloads. JWT Bearer token in Authorization header. BFF validates token before routing.
②
BFF → Middleware: Internal REST over Docker network. No TLS needed (container-to-container). Service discovery via Docker DNS (http://middleware:8082).
③
Middleware → System API: OpenFeign declarative REST client. Automatic serialization/deserialization. Circuit breaker wraps each call. Retry on transient failures.
④
System API → PostgreSQL: JDBC via HikariCP connection pool. JPA/Hibernate handles ORM. Connection URL from environment variable — never hardcoded.

🛡️ Cross-Cutting Concerns (All 3 Layers)

●
Observability: OpenTelemetry agent auto-instruments all 3 services. Distributed traces propagate via traceparent header across BFF → MW → SYS. Jaeger for visualization, Prometheus for metrics.
●
Security: OWASP Top 10 addressed. Input validation at every boundary. SQL injection prevention via parameterized JPA queries. XSS prevention via output encoding. Secrets in environment variables.
●
Health & Readiness: Each service exposes /actuator/health and /actuator/readiness. Docker Compose uses healthcheck to enforce startup order.
●
Config Management: Profile-based YAML files. Docker environment variables override defaults. No secrets in code or Git. 12-Factor compliant.
●
Error Handling: RFC 7807 Problem Detail JSON format across all layers. Consistent error structure: type, title, status, detail, instance.
Bottom-Up
Build Layer 3 (System API) first. Get data flowing. Then Layer 2 (Middleware) to orchestrate. Finally Layer 1 (BFF) to expose to clients.
Contract-First
Define REST API contracts (OpenAPI spec) before writing code. Each layer's API is its contract with the layer above. Break changes = break trust.
Fail-Safe
Every call across layers can fail. Design for failure from Day 1. Circuit breakers, retries, timeouts, and fallback responses are not optional — they're required.

DAY 3 Phase 1 — Foundation

Hexagonal Architecture & Clean Package Structure

📥 Input (What you start with)
Day 2 — 3 running Spring Boot apps with Actuator health Flat/default package structure (single package per module) No internal architecture yet — just @SpringBootApplication classes
🔧 Activities (Step-by-step)
  1. Understand Hexagonal Architecture — Study Ports & Adapters pattern. The domain (business logic) sits at the centre. Inbound adapters (REST controllers) call domain ports. Outbound adapters (JPA repos, Feign clients) implement domain ports. Dependencies flow inward only.
  2. Create package structure in system-api — Organize into 5 top-level packages: adapter.in.web, adapter.out.persistence, application.port.in, application.port.out, and domain.model. This is the template for all 3 modules.
  3. Create package structure in middleware — Same pattern but outbound adapter is adapter.out.feign (calls to system-api via OpenFeign) instead of persistence. Add adapter.in.web for its own REST controllers.
  4. Create package structure in bff-gateway — Lighter: config (gateway routes, security), filter (custom gateway filters), aggregation (response composers). Gateway is reactive — less traditional hex layers.
  5. Define Port interfaces — In system-api, create CreateProductUseCase (inbound port interface) and LoadProductPort (outbound port interface). Write them as plain Java interfaces — no Spring annotations.
  6. Wire with Spring DI — Create a @Service class that implements inbound ports and uses injected outbound ports. The controller calls the inbound port, not the service directly. The service calls the outbound port, not the repository directly.
  7. Enforce dependency rules — Verify: domain.model imports ZERO Spring classes. application.port.* imports ZERO adapter classes. Only adapters import Spring/JPA/Feign. Use ArchUnit to codify this.
  8. Add ArchUnit dependency — Add com.tngtech.archunit:archunit-junit5 to the parent POM. Write a single architecture test that fails if domain imports adapter classes.
📐 Principles Applied
  • SOLID (D) — Dependency Inversion: high-level domain depends on abstractions (ports), not low-level adapters (JPA, Feign).
  • SOLID (I) — Interface Segregation: separate inbound port per use case (CreateProductUseCase, GetProductUseCase) — not one fat interface.
  • SOLID (O) — Open/Closed: add new adapters (e.g., gRPC) without changing domain. Just implement the port.
  • Clean Architecture — Domain has zero framework dependencies. Testable in isolation with plain JUnit — no Spring context needed.
  • Hexagonal / Ports & Adapters — Application core is framework-agnostic. Swap PostgreSQL for MongoDB by changing only the outbound adapter.

💻 Port & Adapter Wiring
// ── Inbound Port (interface) ──
public interface CreateProductUseCase {
    Product create(CreateProductCommand cmd);
}

// ── Outbound Port (interface) ──
public interface SaveProductPort {
    Product save(Product product);
}

// ── Domain Service (implements inbound, uses outbound) ──
@Service
@RequiredArgsConstructor
public class ProductService
        implements CreateProductUseCase {

    private final SaveProductPort savePort;  // outbound

    @Override
    public Product create(CreateProductCommand cmd) {
        Product product = Product.from(cmd);  // domain logic
        return savePort.save(product);
    }
}

// ── Outbound Adapter (implements outbound port) ──
@Component
@RequiredArgsConstructor
public class ProductPersistenceAdapter
        implements SaveProductPort {

    private final ProductJpaRepository repo;

    @Override
    public Product save(Product product) {
        ProductJpaEntity entity = toEntity(product);
        return toDomain(repo.save(entity));
    }
}

💻 Dockerfile (system-api)
# ── Stage 1: Build ──
FROM eclipse-temurin:21-jdk AS build
WORKDIR /workspace
COPY pom.xml .
COPY system-api/pom.xml system-api/
# Cache dependencies
RUN mvn dependency:go-offline -pl system-api -am
COPY system-api/src system-api/src
RUN mvn clean package -pl system-api -DskipTests

# ── Stage 2: Runtime ──
FROM eclipse-temurin:21-jre-alpine
RUN addgroup -S app && adduser -S app -G app
WORKDIR /app
COPY --from=build /workspace/system-api/target/*.jar app.jar
USER app
EXPOSE 8081
ENTRYPOINT ["java", \
  "-XX:+UseZGC", \
  "-XX:MaxRAMPercentage=75.0", \
  "-jar", "app.jar"]

@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    ProblemDetail handleValidation(
            MethodArgumentNotValidException ex) {
        ProblemDetail pd = ProblemDetail.forStatus(400);
        pd.setTitle("Validation Failed");
        pd.setType(URI.create("/errors/validation"));
        List<Map<String,String>> errors = ex.getFieldErrors()
            .stream()
            .map(e -> Map.of(
                "field", e.getField(),
                "message", e.getDefaultMessage()))
            .toList();
        pd.setProperty("errors", errors);
        return pd;
    }

    @ExceptionHandler(EntityNotFoundException.class)
    ProblemDetail handleNotFound(EntityNotFoundException ex) {
        ProblemDetail pd = ProblemDetail.forStatus(404);
        pd.setTitle("Resource Not Found");
        pd.setDetail(ex.getMessage());
        return pd;
    }
}

@RestControllerAdvice
@Slf4j
public class MiddlewareExceptionHandler {

    @ExceptionHandler(FeignException.class)
    ProblemDetail handleFeign(FeignException ex) {
        int status = switch (ex.status()) {
            case 400 -> 400;
            case 404 -> 404;
            default  -> 502; // Bad Gateway
        };
        ProblemDetail pd = ProblemDetail.forStatus(status);
        pd.setTitle("Downstream Service Error");
        pd.setProperty("correlationId", MDC.get("correlationId"));
        return pd;
    }

    @ExceptionHandler(CallNotPermittedException.class)
    ProblemDetail handleCircuitOpen(CallNotPermittedException ex) {
        ProblemDetail pd = ProblemDetail.forStatus(503);
        pd.setTitle("Service Temporarily Unavailable");
        pd.setDetail("Circuit breaker is OPEN for: " +
                      ex.getCausingCircuitBreakerName());
        return pd;
    }
}

@Component
@Slf4j
public class RequestTimingFilter implements GlobalFilter, Ordered {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange,
                              GatewayFilterChain chain) {
        long start = System.currentTimeMillis();
        String path = exchange.getRequest().getPath().value();
        return chain.filter(exchange).then(Mono.fromRunnable(() -> {
            long ms = System.currentTimeMillis() - start;
            exchange.getResponse().getHeaders()
                .add("X-Response-Time", ms + "ms");
            log.info("Gateway: {} {} → {}ms", 
                exchange.getRequest().getMethod(), path, ms);
        }));
    }

    @Override public int getOrder() { return -1; }
}

Saga Pattern — Orchestrated Distributed Transactions

📥 Input
Day 23 — Kafka events flowing between services No compensation logic when downstream steps fail
🔧 Activities
  1. Design order saga — Steps: Create Order → Reserve Inventory → Process Payment → Confirm Order. Each step has a compensating action (Cancel Order, Release Inventory, Refund Payment).
  2. Saga orchestrator — Build OrderSagaOrchestrator in middleware. Listens for events, decides next step. Maintains saga state in DB (STARTED, INVENTORY_RESERVED, PAYMENT_PROCESSED, COMPLETED, COMPENSATING, FAILED).
  3. Compensation flow — If payment fails → publish ReleaseInventoryCommand → publish CancelOrderCommand. Each service handles its own rollback. Eventually consistent.
  4. Saga state machine — Use Spring State Machine or simple enum-based FSM. Track state transitions. Log every transition for audit.
  5. Timeout handling — If no response within 30s → trigger compensation. Use Kafka consumer timeout + scheduled job to check stale sagas.
  6. Testing sagas — Happy path: all steps succeed. Failure at step 2: compensation triggered. Failure at compensation: alert + manual intervention queue.
  7. Saga observability — Custom metrics: saga_started, saga_completed, saga_compensated. Trace spans for each saga step. Dashboard showing saga success rate.
📐 Principles
  • BASE over ACID — Basically Available, Soft state, Eventually consistent. No distributed locks.
  • Compensating transactions — Every action must have an undo. Design for failure from the start.
  • Orchestration vs Choreography — Orchestrator = centralized control. Choreography = each service reacts. We use orchestration for complex multi-step flows.
💻 Saga Orchestrator
@Service
@RequiredArgsConstructor
public class OrderSagaOrchestrator {

    private final SagaStateRepo repo;
    private final KafkaTemplate<String, Object> kafka;

    @KafkaListener(topics = "order-events")
    public void onOrderCreated(OrderCreatedEvent e) {
        var saga = SagaState.start(e.orderId());
        repo.save(saga);
        kafka.send("inventory-commands",
            new ReserveStockCmd(e.orderId(),
                e.items()));
        saga.transition(INVENTORY_PENDING);
        repo.save(saga);
    }

    @KafkaListener(topics = "inventory-events")
    public void onInventoryResult(
            InventoryEvent event) {
        var saga = repo.findByOrderId(
            event.orderId());
        if (event instanceof InventoryReserved) {
            saga.transition(INVENTORY_OK);
            kafka.send("payment-commands",
                new ProcessPaymentCmd(
                    saga.orderId(), saga.total()));
        } else {
            saga.transition(COMPENSATING);
            kafka.send("order-commands",
                new CancelOrderCmd(saga.orderId()));
        }
        repo.save(saga);
    }
}

Performance Testing — k6 Load Tests & Optimization

📥 Input
Day 27 — Full CI/CD pipeline deploying to K8s No performance baselines, unknown capacity limits
🔧 Activities
  1. Install k6 & write test scripts — JavaScript-based load test scenarios. Test order creation flow: authenticate → create order → verify response. Parameterize with test data from CSV.
  2. Smoke test (1 VU) — Single virtual user, verify correctness under no load. Establish baseline response times and validate test script logic.
  3. Load test (50 VUs, 5 min) — Simulate normal traffic. Measure p50/p95/p99 latency, throughput (RPS), error rate. Set SLA thresholds: p99 < 500ms, error rate < 1%.
  4. Stress test (ramp to 200 VUs) — Find breaking point. Ramp up gradually: 10 → 50 → 100 → 200 VUs. Monitor where latency spikes and errors increase. Identify bottleneck.
  5. Bottleneck analysis — Correlate k6 results with Grafana dashboards. Check: connection pool exhaustion? Thread starvation? GC pauses? DB query slow? Kafka consumer lag?
  6. Optimization cycle — Add database indices, tune connection pool sizes (hikari.maximum-pool-size), enable response compression, add Redis cache for hot paths. Re-test.
  7. Performance CI gate — Add k6 to CI pipeline. Run smoke test on every PR. Run full load test nightly. Fail if SLA thresholds breached. Track trends over time.
📐 Principles
  • Test early, test often — Performance is a feature. Regressions caught by automated load tests in CI.
  • Little's Law — L = λW (concurrency = throughput × latency). Understand system capacity mathematically.
  • Amdahl's Law — Optimize the bottleneck first. Profiling before guessing.

import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate, Trend } from 'k6/metrics';

const errorRate = new Rate('errors');
const orderLatency = new Trend('order_latency');

export const options = {
  stages: [
    { duration: '1m', target: 10  },
    { duration: '3m', target: 50  },
    { duration: '2m', target: 100 },
    { duration: '1m', target: 0   },
  ],
  thresholds: {
    http_req_duration: ['p(99)<500'],
    errors: ['rate<0.01'],
  },
};

const BASE = 'http://bff-gateway:8080';

export default function () {
  // 1. Authenticate
  const auth = http.post(
    `${BASE}/auth/token`,
    JSON.stringify({
      username: 'testuser',
      password: 'testpass' }),
    { headers:
      { 'Content-Type': 'application/json' }}
  );
  const token = auth.json('access_token');

  // 2. Create Order
  const start = Date.now();
  const res = http.post(
    `${BASE}/api/v1/orders`,
    JSON.stringify({
      customerId: 'cust-001',
      items: [{ sku: 'SKU-100', qty: 2 }]
    }),
    { headers: {
        'Content-Type': 'application/json',
        Authorization: `Bearer ${token}`
    }}
  );
  orderLatency.add(Date.now() - start);

  check(res, {
    'status 201': (r) => r.status === 201,
    'has orderId': (r) =>
      r.json('orderId') !== undefined,
  }) || errorRate.add(1);

  sleep(1);
}

# ADR-001: Hexagonal Architecture
## Status: Accepted
## Context
We need a service architecture that
isolates business logic from infrastructure
to enable testability, portability, and
independent evolution of adapters.

## Decision
Adopt Hexagonal (Ports & Adapters) for
all microservices. Domain logic in core
module. REST/Kafka/DB as adapters.

## Consequences
+ Business logic testable without Spring
+ Swappable adapters (REST → gRPC)
+ Enforced by ArchUnit in CI
- Steeper learning curve initially
- More packages/classes than layered
